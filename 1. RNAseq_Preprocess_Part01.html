Introduction to RNAseq Analysis: Part One ¶

Overview¶

The next generation sequencing (NGS) data in this example was obtained from NCBI GEO and the Sequence Read Archive (SRA). This notebook is not intended to show you how to do these steps yourself. Attempting such a project requires at least a moderate level of experience with computing in Unix/Linux. However, we will provide at least a brief overview of each step so that you understand how NGS data is handled. Importantly, there are always new developments in tools and algorithms to analyze NGS data, so the pipeline shown here is only one of many possible pipelines.

Here are the steps that we will cover in this Notebook:

  1. Accessing SRA data.
  2. QC of reads pre-trimming with FastQC.
  3. Quality trimming of sequence reads with Trimmomatic.
  4. QC of reads post-trimming with FastQC.
  5. Sequence read alignment with STAR.
  6. Overview of BAM file format.
  7. Generating gene count matrix.
Please note that most preprocessing analyses and tools used in this notebook need to be run in bash shell. The programming language is denoted at the beginning of the code cell using either `%use bash` or `%use r`.

FASTQ Files¶

FASTQ files are plain text files that contain your raw NGS data. Each read requires four lines in a FASTQ file:

  1. Line 1 begins with @ followed by the read identifier and optional description.
  2. Line 2 is the read sequence in capital letters.
  3. Line 3 begins with + followed by the read identifier (Line 1).
  4. Line 4 contains the sequence base quality scores encoded as an ASCII character. There must be a quality score for every base in the sequence, i.e. Line 2 and Line 4 must be the same length.

We can take a peek at one of our FASTQ files.

Read Quality Control and Trimming¶

There are multiple steps required to produce a sample-specific FASTQ file from a sequencing run. Remember, the primary signal to determine a DNA sequence is a light signal, so there is software for that. Different samples are usually run in the same lane using indexes (multiplexed), so there is software that de-multiplexes the reads by detecting the index sequences. When RNA or DNA is prepared for sequencing, adaptors are added to the ends (as are indexing sequences), so these need to be removed. Finally, sequences at the end of a run often have lower quality scores (more uncertainty in base calling), so these low quality bases are usually trimmed. This process can actually remove entire reads if trimming produces a read that is too short.

The example data was downloaded from SRA, and it is surprising how vague the data processing details are. We have sample-specific data, so clearly the sequences have been de-multiplexed, but how much trimming has been done? Are the adaptors removed? Are the sequences quality trimmed? There is a hint in the FASTQ files in that the length of every sequence is 101 bases. That suggests that no quality score-specific trimming was done because the number of trimmed bases would vary.

If you are new to NGS experiments, we strongly suggest that you work with your genomics or bioinformatics core to process and QC your data at least to the alignments stage. Make sure that you request the details of every step in the data processing such as:

  1. Reads per flowcell to determine if there were any anomolies for the run.
  2. Total reads for each sample to determine how well the libraries were balanced.
  3. Read quality report before and after trimming to determine quality of sequences.
  4. Trimming program and all arguments or parameters to document your analysis.
  5. Reads remaining after trimming.
    a. For paired end runs, number of paired reads.
    b. Number of unpaired first reads.
    c. Number of unpaired second reads.
  6. All parameters and arguments for the aligning procedure, e.g. software, parameters and genome assembly.
  7. Number of mapped reads.
    a. Number of reads that map as proper or concordant pairs (details below).
    b. Number of reads that map in improper or discordant pairs.
    c. Number of reads that map unpaired.
    d. Number of unmapped reads.

A good review of NGS data problems is provided by Zhou and Rokas, 2014. There will be additional details below.

Sequence Read Quality Reports¶

Modern high throughput sequencers can generate tens of millions of sequences in a single run. Before analysing this sequence to draw biological conclusions you should always perform some simple quality control checks to ensure that the raw data looks good and there are no problems or biases in your data which may affect how you can usefully use it.

Most sequencers will generate a QC report as part of their analysis pipeline, but this is usually only focused on identifying problems which were generated by the sequencer itself. FastQC from Babraham Bioinformatics provides a QC report of FASTQ files which can spot problems which originate either in the sequencer or in the starting library material.

Trimming Reads¶

The FastQC reports make it clear that we need to perform quality trimming of our reads. There are many tools to help you with this (see Zhou and Rokas). Importantly, you should remove adaptors first, then trim for sequence quality because even low quality sequences can be used to help identify adaptors.

Be cautious with the tool that you choose. Some programs do not handle paired-end data well. They produce a single output file for each input file, and there is no cross-checking between the files for the first and second reads. The reason this matters is that some alignment programs assume that the first and second reads are paired, AND IN THE SAME ORDER IN THE FASTQ FILES. If a trimming program drops even a single first or second read due to quality problems, this is no longer true, and the resulting alignment will be nonsense. The majority of alignments will be discordant, because the first and second reads will often align to different chromosomes, the same strand or too many bases apart.

We chose Trimmomatic to trim our reads because it can search for adaptors and low quality sequences in the same run. For paired end data, Trimmomatic accepts both the first and second read FASTQ file, and produces four output files:

  1. First Read Paired
  2. First Read Unpaired
  3. Second Read Paired
  4. Second Read Unpaired

The paired files could then be aligned as paired-end reads. In some cases, you can choose to align the unpaired reads as single end data.

An important point here is choice of genome assembly. The human genome assembly continues to improve with improved sequencing and assembly methods. Here are the most common human genome assemblies in reverse chronological order.

Running STAR Aligner¶

There are many alignment programs to choose from. We need an aligner that can handle spliced alignments that result from RNA splicing. We also need an aligner that can handle paired end reads. STAR is an ultrafast universal RNA-seq aligner that performs very well in bench marking tests. If you are interested in learning more, see the STAR article in Bioinformatics or the STAR Github repository.

We already have STAR software installed on our server. You will need to load the STAR software with module load before running, and file names and paths need to be changed accordingly. To download and install STAR on your own computer, please follow instruction in the STAR Github repository.

Before alignment using STAR, the first step is to generate genome indices for the reference genome. This step is done by specifying the --runMode option to be genomeGenerate. We will then use the generated indices to align the reads by specifying the --runMode option to be alignReads.

The parameters for generating genome indices are listed below:

  • --runThreadN specifies the number of threads (processors) to run STAR
  • --runMode genomeGenerate directs STAR to generate genome indices
  • --genomeDir specifies the path to store the output genome indices
  • --genomeFastaFiles specifies the path to the annotation FASTA file
  • --sjdbGTFfile specifies the path to the GTF file with gene annotations
  • --sjdbOverhang maximum possible overhang for your reads

The human GRCh38 annotation FASTA file can be downloaded from Ensembl FTP download website.

The important parameters for alignReads are listed below.

  • --runThreadN specifies the number of threads (processors) to run STAR
  • --runMode alignReads directs STAR to align reads
  • --genomeDir specifies the path to the genome directory where genome indices where generated
  • --runDirPerm specifies the permission for the directories created at the run-time. All_RWX means "all read/ write/execute access"
  • --sjdbGTFfile specifies the path to the GTF file with gene annotations
  • --readFilesIn specifies the paths to files that contain input read1 (and, if needed, read2)
  • --outSAMtype specifies the file type for output. SortedByCoordinate will output the sorted file, which is similar as samtools sort command.
  • --outFileNamePrefix specifies prefix part of the file output name

Below is the command that aligns one pair of files for sample SRR3437043. This time we will use the alignReads mode in STAR. This command is broken down into multiple lines to make it more legible. For project data, you can run it by changing the data_path and out_path to another local folder.

A special note is that because STAR doesn't handle Gzip reads files directly, we need to unzip the trimmed data from last step using gunzip and read the unzipped file in --readFilesIn. The -c option in gunzip will read in the extracted file without replacing the original gzip file, which can save some storage.

Gene Annotations¶

There are many ways to obtain a file of gene annotations. Common sources are General Feature Format (GFF) files that contain genome information, or Gene Transfer Format (GTF) files that contain information on the structure of genes. These files are available from sources such as NCBI, UCSC and Ensembl. In addition, Bioconductor maintains databases for some organisms that can also be used.

Regardless of the source of gene annotation, it is critical that the genes annotation includes exon structure, especially for mammalian genomes, because mRNAs arise from transcribed exons, not the intervening introns. For this project, we downloaded the annotation from iGenomes. Again, be careful, the annotation file that you choose must be consistent with the genome assembly used for the alignments!

In the rest of this section, we are going to convert a Gene Transfer File (GTF) to a GRangesList that can be used to quantify gene expression from BAM files.

Important Factors for Gene Counts¶

We are almost ready to get our gene counts, but there are several other factors to consider,

  1. Do you have a strand specific library?
  2. Did you do paired-end sequencing? If so, which end will map to your gene?
  3. How stringent do you want the counting mode to be?

Strand-Specific Libraries¶

Today, most RNA sequencing is strand-specific. However, you must confirm that this is true. A strand-specific library allows you to resolve overlapping genes on opposite strands. It can also allow you to detect antisense RNAs. But, if you have paired-end reads, you will have reads that map to both strands of the gene. Either way, you should confirm how your reads map to the gene. You can try to track this information down, but it is a simple matter to look at your alignments in IGV to confirm this.

In the image above, the gene on the left is on the minus strand while the gene on the right is on the plus strand. In IGV, I right clicked on the alignments and selected, Color alignments by > first-of-pair strand. Red indicates plus strand while blue indicates minus strand. The interpretation: the first read sequenced maps to the opposite strand of the gene. We need to remember this when we quantify gene expression.

Paired End Libraries¶

Software that determine gene counts from alignments will generally have a paired-end mode. Remember, you cannot count both reads from a single alignment because they originate from the same RNA molecule. In this Notebook, we will actually use the single-end mode but use a filter so that only first-in-pair reads are used.

Counting Modes¶

The stringency of counting is also determined by the counting mode that you choose. The counting modes in Bioconductor are the same as the command line program HTseq. Below is a figure that used to accompany the HTseq user manual.

The above illustration applies to non-strand specific counting. In the last example, if Feature 1 was on the + strand and Feature 2 on the - strand. The Union mode could resolve this as long as the read mapped to Feature 1. The mode that you choose should be determined by your library construction and the confidence in the gene annotation. For example, you would not IntersectionStrict if you had a high proportion of unprocessed transcripts. For more information about counting modes, you can read the vignette for summarizeOverlaps.

Usage of HTSeq: htseq-count [options] <alignment_files> <gtf_file>

Explaination of commands:

  • positional arguments:

    alignment_files: Path to the SAM/BAM files containing the mapped reads. If '-' is selected, read from standard input

    gtf_file: Path to the GTF file containing the features

  • Options:

    -f {sam,bam,auto}: Type of data. DEPRECATED: file format is detected automatically. This option is ignored.

    -r {pos,name}: 'pos' or 'name'. Sorting order of (default: name).

    -t FEATURE_TYPE: Feature type (3rd column in GTF file) to be used, all features of other type are ignored (default, suitable for Ensembl GTF files: exon)

    -i IDATTR: GTF attribute to be used as feature ID (default, suitable for Ensembl GTF files: gene_id). The typical way of using this option is to count all exonic reads from each gene and add the exons but other uses are possible as well.
